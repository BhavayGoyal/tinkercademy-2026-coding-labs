Q1 (DRY — Part 3).
    Quote the exact lines or expressions that were duplicated between your
    Part 2 copy constructor and copy assignment. Which of the two refactors
    did you choose (helper method, or copy-and-swap), and how does it
    eliminate that duplication? If you chose copy-and-swap, also explain
    why it is exception-safe.

YOUR ANSWER:
The exact duplicate lines were:
_size = other._size;
_capacity = other._capacity;
_name = other._name;
_friends = new std::string[_capacity];
for (size_t i = 0; i < _size; ++i) {
    _friends[i] = other._friends[i];
}
I chose copy-and-swap refactor as it eliminates duplication by having the copy assignment operator take its parameter by value (which calls the copy constructor automatically) and then swapping
the current object's state with that temporary copy. This way, the deep-copy logic only exists in the copy constructor and is being done once only.
It is exception-safe also because all the dangerous operations like memory allocation or string copying will happen while creating the temporary copy. If exception occurs, the original object's
state hasn't been changed yet.

Q3 (Rule of Zero — Part 6, if attempted).
    Why does removing every SMF declaration from your `FriendList` and
    `User` work after Part 6, but would have been a disaster after Part 1?
    Specifically: in Part 2 you deleted the move SMFs on `User`. Do you
    still want them deleted now that you've reached Part 6? Argue one way
    or the other in 2–4 sentences.

YOUR ANSWER:
In part 1, the friend list is stored as a manually managed heap-allocated std::string array while in part 6, friends list is stored as an automatically managed std::vector of strings.
Because all members of User (string and vector) are RAII and manage their own resources, User can follow the Rule of Zero and define no SMFs at all.
Regarding the move SMFs: We do not need to delete them, but in part 2 they were deleted because we had not written logic to safely move the raw pointers so to maintain the consistent answers (and also the test harness requires them to be deleted)
I marked them as delete in part 6 too. However, std::vector is indeed designed to move efficiently, so if we allow the compiler to generate move SMFs, we can allow the compiler to move User 
objects.
